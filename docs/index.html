⚡ 개발 워크플로우

게임 개발의 전체 라이프사이클을 단계별로 지원하는 워크플로우입니다.

🗺️ 개발 단계 흐름

1
사전 제작 (Pre-production)
브레인스토밍 → 게임 브리프 작성
게임 컨셉 아이디어를 발산하고, 핵심 비전을 문서화합니다.
2
설계 (Design)
GDD 작성 → 내러티브 디자인
게임 메카닉, 시스템, 진행 구조와 스토리를 상세하게 설계합니다.
3
기술 설계 (Technical)
프로젝트 컨텍스트 → 게임 아키텍처 → 테스트 프레임워크
기술 스택, 시스템 아키텍처, 테스트 인프라를 구축합니다.
4
제작 (Production)
스프린트 계획 → 스토리 생성 → 개발 → 코드 리뷰 → 회고
실제 구현, 테스트, 배포까지의 반복 개발 사이클을 운영합니다.

🚀 빠른 워크플로우 (언제든지 사용 가능)

⚡
Quick Spec
대화형 탐색을 통해 구현 준비가 완료된 기술 사양서를 빠르게 작성합니다.
🔨
Quick Dev
기술 스펙 또는 직접 지시사항을 기반으로 유연한 개발 워크플로우를 실행합니다.
🧪
Quick Prototype
게임 메카닉과 아이디어를 빠르게 프로토타이핑하여 검증합니다.
📄
Document Project
기존 프로젝트를 분석하여 유용한 문서를 자동으로 생성합니다.

💬 스킬 명령어

"+" → "플러그인" → "Game dev studio"에서 선택하거나, 자연어로 요청할 수 있습니다.

명령어 설명 단계
create-game-brief단계별 협업 탐색을 통해 게임 비전을 정의하고 게임 브리프 문서를 작성합니다.사전 제작
brainstorm다양한 창의적 기법과 아이디어 발상 방법을 활용한 인터랙티브 브레인스토밍 세션을 진행합니다.사전 제작
create-gdd협업 탐색을 통해 포괄적인 게임 디자인 문서(GDD)를 작성합니다.설계
generate-project-contextAI 에이전트의 일관된 동작을 위한 최적화된 프로젝트 컨텍스트 파일을 생성합니다.기술
quick-spec대화형 탐색을 통해 구현 준비가 완료된 기술 사양서를 작성합니다.전체
quick-dev기술 스펙 또는 직접 지시사항을 기반으로 유연한 개발 워크플로우를 실행합니다.전체
helpGame Dev Studio 워크플로우에서 다음 단계를 분석하고 안내합니다.전체
party-mode모든 Game Dev Studio 에이전트 간의 그룹 토론을 조율하는 멀티 에이전트 대화를 시작합니다.전체
index-docs대상 폴더의 모든 문서를 참조하는 index.md를 생성하거나 업데이트합니다.전체
advanced-elicitationLLM이 최근 출력을 재검토하고 정제하여 개선하도록 고급 도출 기법을 적용합니다.전체
review-adversarial콘텐츠에 대한 비판적 리뷰를 수행하고 최소 10개의 문제점을 찾아 보고합니다.전체
review-edge-cases모든 분기 경로와 경계 조건을 탐색하여 처리되지 않은 엣지 케이스만 보고합니다.전체
review-prose텍스트의 커뮤니케이션 문제를 검토하고 3열 표로 수정 제안을 출력합니다.전체
review-structure이해도를 유지하면서 삭제, 재구성, 단순화를 제안하는 구조적 편집 리뷰를 수행합니다.전체
shard-doc대용량 마크다운 문서를 레벨 2 섹션 기준으로 더 작은 파일들로 분할합니다.전체

💡 팁 & FAQ

자주 묻는 질문과 효과적인 사용 팁입니다.

?
어떤 게임 엔진을 지원하나요?
▼

Unity, Unreal Engine, Godot 및 기타 엔진을 지원합니다. 설치 시 선택한 플랫폼에 맞춰 워크플로우가 최적화됩니다.

?
처음 시작할 때 어떤 워크플로우부터 사용하면 좋을까요?
▼

추천 순서:

1. brainstorm — 게임 아이디어 발산

2. create-game-brief — 게임 비전 정의

3. create-gdd — 상세 디자인 문서 작성

4. generate-project-context — 프로젝트 컨텍스트 생성

5. quick-dev — 구현 시작

💡 작은 프로젝트라면 quick-spec → quick-dev로 바로 시작해도 됩니다.
?
party-mode는 무엇인가요?
▼

모든 전문 에이전트가 하나의 대화에 참여하여 그룹 토론을 진행합니다. 각 에이전트가 자신의 전문 분야 관점에서 의견을 제시합니다.

예를 들어 "이 게임 컨셉이 기술적으로 가능한가?"라는 질문에 아키텍트, 개발자, QA, 디자이너가 각각 의견을 냅니다.

?
help 명령어는 언제 사용하나요?
▼

현재 프로젝트 상태를 분석하고 다음에 무엇을 해야 하는지 안내받을 때 사용합니다.

"다음에 뭘 해야 하지?" 또는 "지금 어떤 단계야?"라고 물어보면 됩니다.

?
플러그인 업데이트는 어떻게 하나요?
▼

현재 Cowork의 자동 업데이트에 알려진 버그가 있어, 다음 방법을 권장합니다:

1. 플러그인 관리에서 game-dev-studio 삭제

2. 다시 마켓플레이스에서 재설치

⚠️ 재설치해도 프로젝트 데이터나 생성된 문서에는 영향이 없습니다. 플러그인 자체만 갱신됩니다.
?
출력 파일은 어디에 저장되나요?
▼

기본 출력 경로:

{프로젝트 루트}/_bmad-output/

세부 경로:

• 기획 문서: _bmad-output/planning-artifacts/

• 구현 결과물: _bmad-output/implementation-artifacts/

• 프로젝트 문서: docs/

?
채팅 탭과 Cowork 탭의 차이는 무엇인가요?
▼

채팅 — 일반적인 Claude AI 대화 (질문, 글쓰기, 코드 등)

Cowork — 플러그인과 워크플로우를 사용한 구조화된 작업 수행. 프로젝트 폴더 연결, 스킬 실행 등이 가능합니다.

Game Dev Studio 플러그인은 Cowork 탭에서 사용합니다.
